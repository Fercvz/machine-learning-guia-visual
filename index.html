Machine learning para o dia a dia

Pipeline completovisual explicativo · animação leve

Dados viram decisão por uma sequência repetível — o algoritmo é apenas uma parte.

Este é um manual de campo para transformar problemas de supermercado, construção civil e varejo em sistemas úteis. A rota principal mostra o que aparece no trabalho diário; conteúdos raros ficam recolhidos como referência.

decisão → dados → ação → valor matemática conectada à decisão complexidade somente quando necessária
1 · Defina a decisãoPrever, classificar, agrupar ou otimizar?
2 · Valide em dados novosDesempenho no treino não prova generalização.
3 · Meça a operaçãoA previsão precisa mudar uma ação e gerar valor.

O ciclo de vida que orienta projetos responsáveis

  1. Dado bruto — vendas do PDV, planilha de orçamento de obra, histórico de compras do cliente.
  2. Features — as colunas que realmente importam (dia da semana, m² da obra, ticket médio).
  3. Modelo — a "fórmula" que aprende o padrão entre as features e o resultado.
  4. Treino — o modelo ajusta seus parâmetros olhando exemplos passados.
  5. Avaliação — você mede se o modelo acerta em dados que ele nunca viu.
  6. Predição — usar o modelo pronto em dados novos (a obra que ainda não terminou, o cliente que ainda não comprou).
Regra de ouro: escolher o algoritmo é apenas uma etapa. O projeto precisa definir o instante da previsão, impedir vazamento, comparar uma baseline, ligar a saída a uma ação e medir valor depois do deploy.

A definição que resume tudo

"O aprendizado de máquina tem a ver com a construção de sistemas que podem aprender com os dados. Aprender significa melhorar em alguma tarefa, considerando alguma medida de desempenho."

Essa frase parece abstrata, mas ela já contém as três perguntas que você precisa responder em qualquer projeto, antes de pensar em modelo:

  • Qual é a tarefa? Ex.: prever o custo de uma obra, decidir se um cliente vai cancelar, agrupar fornecedores parecidos.
  • Onde estão os dados que representam essa tarefa? Ex.: histórico de obras concluídas, histórico de compras do cliente, cadastro de fornecedores.
  • Qual é a medida de desempenho? Ex.: erro médio em R$ na estimativa de custo (folha 18), F1-score na previsão de churn (folha 18) — isso é literalmente a função de avaliação que você já viu na folha 17.
Por que isso importa: se você não consegue responder as três perguntas acima pro seu problema, ainda não é hora de escolher modelo — é hora de definir melhor a tarefa e achar a medida de desempenho certa. A maioria dos projetos de ML que falham, falha aqui, não na escolha do algoritmo.

O trabalho real de Machine Learning

O modelo ocupa uma parte pequena do projeto. O trabalho diário é transformar uma decisão ambígua em dados auditáveis, uma avaliação honesta, uma ação executável e um resultado monitorado.

Anti-padrões que parecem progresso

Escolher XGBoost antes do alvo

Complexidade não corrige uma decisão ou rótulo mal definido.

Split aleatório em dados temporais

O modelo aprende com um futuro que não existirá no deploy.

Celebrar acurácia sem baseline

Um número isolado não prova melhoria sobre a operação atual.

Entregar probabilidade sem ação

Score sem limiar, capacidade e responsável vira um dashboard sem consequência.

Monitorar somente drift

Mudança nas entradas não substitui medir resultado e impacto real.

Retreinar por calendário

Retreine por evidência de degradação, mudança relevante ou nova cobertura.

Quando não usar Machine Learning

  • Uma regra determinística resolve com segurança.
  • Não existe ação após a previsão.
  • Há poucos exemplos e nenhum processo para obter rótulos.
  • O evento muda antes de o modelo chegar à produção.
  • O custo de erro é inaceitável sem revisão humana.
  • O ganho potencial não paga integração e manutenção.

Unidade de análise

Uma linha deve representar algo explícito: produto-loja-dia, cliente-mês ou obra no marco de 30% de execução. Misturar unidades torna a métrica ambígua.

Tempo de predição

Defina quando a previsão é feita e remova qualquer variável criada depois desse instante. Isso evita target leakage.

Horizonte e alvo

“Demanda” é vago. “Unidades vendidas nos próximos 7 dias, sem censura por ruptura” é um alvo verificável.

Ação e custo

Uma probabilidade não é uma decisão. Estoque, inspeção ou contato dependem de capacidade, margem e custo de cada erro.

Auditar e reconstruir demanda

Venda observada pode ser menor que demanda quando o produto acabou. Marque períodos censurados; não ensine “estoque zero = baixa procura”.

Criar features sem futuro

Validar no tempo

70% treino, 15% validação, 15% teste. Compare com “mesmo dia da semana anterior”.

Transformar previsão em pedido

Demanda durante o lead time + estoque de segurança − posição atual, arredondada ao lote.

Monitorar por segmento

Pipeline executável

import joblib
import numpy as np
import pandas as pd
from sklearn.compose import ColumnTransformer
from sklearn.ensemble import HistGradientBoostingRegressor
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import OneHotEncoder

df = pd.read_csv("vendas_diarias.csv", parse_dates=["data"])
keys = ["loja_id", "produto_id"]
df = df.sort_values(keys + ["data"]).copy()

# Toda estatística termina antes do dia previsto.
g = df.groupby(keys, observed=True)["vendas"]
for lag in (7, 14, 28):
    df[f"vendas_lag_{lag}"] = g.shift(lag)
df["media_7"] = g.transform(lambda s: s.shift(1).rolling(7).mean())
df["media_28"] = g.transform(lambda s: s.shift(1).rolling(28).mean())
df["dia_semana"] = df["data"].dt.dayofweek
df["mes"] = df["data"].dt.month
df["alvo_d7"] = g.shift(-7)

features_num = ["preco", "promocao", "estoque_fim", "prazo_fornecedor",
                "vendas_lag_7", "vendas_lag_14", "vendas_lag_28",
                "media_7", "media_28", "dia_semana", "mes"]
features_cat = ["loja_id", "produto_id"]
features = features_num + features_cat
data = df.dropna(subset=features + ["alvo_d7"]).copy()

# Cortes cronológicos globais: o teste representa o futuro.
datas = np.sort(data["data"].unique())
c1, c2 = datas[int(.70 * len(datas))], datas[int(.85 * len(datas))]
train = data[data.data < c1]
valid = data[(data.data >= c1) & (data.data < c2)]
test  = data[data.data >= c2]

prep = ColumnTransformer([
    ("cat", OneHotEncoder(handle_unknown="ignore", sparse_output=False), features_cat),
    ("num", "passthrough", features_num),
])
model = HistGradientBoostingRegressor(loss="poisson", max_iter=250,
                                      learning_rate=.06, max_leaf_nodes=31,
                                      l2_regularization=1.0, random_state=42)
pipe = Pipeline([("prep", prep), ("model", model)])
pipe.fit(train[features], train["alvo_d7"])

def metrics(y, pred):
    err = np.asarray(pred) - np.asarray(y)
    return {"WAPE": np.abs(err).sum() / np.abs(y).sum(),
            "BIAS": err.sum() / np.abs(y).sum()}

pred_valid = np.clip(pipe.predict(valid[features]), 0, None)
base_valid = valid["vendas_lag_7"].to_numpy()
print("baseline", metrics(valid["alvo_d7"], base_valid))
print("modelo", metrics(valid["alvo_d7"], pred_valid))

# Segurança baseada somente na validação; teste continua intocado.
safety_q90 = float(np.quantile(np.abs(valid["alvo_d7"] - pred_valid), .90))
train_valid = pd.concat([train, valid])
pipe.fit(train_valid[features], train_valid["alvo_d7"])
pred_test = np.clip(pipe.predict(test[features]), 0, None)
print("teste final", metrics(test["alvo_d7"], pred_test))

artifact = {"model": pipe, "features": features, "safety_q90": safety_q90,
            "trained_until": str(train_valid["data"].max().date())}
joblib.dump(artifact, "demanda_d7.joblib")

Critério de aceite: o modelo precisa vencer a baseline na validação e no teste, manter bias controlado nos maiores segmentos e reduzir ruptura sem inflar excesso de estoque no piloto.

Escala: one-hot de milhares de produtos pode consumir muita memória. Em catálogos grandes, compare codificação temporal segura, modelos com categorias nativas ou modelos globais com embeddings — sempre calculando qualquer estatística apenas no passado.

Pipeline com limiar escolhido por custo

import joblib
import numpy as np
import pandas as pd
from sklearn.compose import ColumnTransformer
from sklearn.impute import SimpleImputer
from sklearn.linear_model import LogisticRegression
from sklearn.metrics import brier_score_loss, precision_score, recall_score, roc_auc_score
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import OneHotEncoder, StandardScaler

df = pd.read_csv("obras_marco_30.csv", parse_dates=["data_marco"])
target = "atraso_maior_14d"
num = ["area_m2", "desvio_fisico_pp", "desvio_custo_pp", "itens_criticos_pendentes",
       "dias_chuva_30d", "experiencia_gestor_anos", "concentracao_fornecedor"]
cat = ["tipo_obra", "regiao", "modalidade_contrato"]
features = num + cat

# Projetos mais novos ficam fora do desenvolvimento.
q70, q85 = df["data_marco"].quantile([.70, .85])
train = df[df.data_marco < q70]
valid = df[(df.data_marco >= q70) & (df.data_marco < q85)]
test = df[df.data_marco >= q85]

prep = ColumnTransformer([
    ("num", Pipeline([("fill", SimpleImputer(strategy="median")),
                      ("scale", StandardScaler())]), num),
    ("cat", Pipeline([("fill", SimpleImputer(strategy="most_frequent")),
                      ("onehot", OneHotEncoder(handle_unknown="ignore"))]), cat),
])
pipe = Pipeline([("prep", prep),
                 ("model", LogisticRegression(max_iter=2000))])
pipe.fit(train[features], train[target])

# O limiar pertence à validação, nunca ao teste.
p_valid = pipe.predict_proba(valid[features])[:, 1]
thresholds = np.linspace(.05, .95, 181)
cost_fn, cost_fp = 15_000, 500
def cost_at(t):
    pred = p_valid >= t
    fn = ((valid[target] == 1) & ~pred).sum()
    fp = ((valid[target] == 0) & pred).sum()
    return fn * cost_fn + fp * cost_fp
threshold = float(min(thresholds, key=cost_at))

p_test = pipe.predict_proba(test[features])[:, 1]
pred_test = p_test >= threshold
print({"threshold": threshold,
       "roc_auc": roc_auc_score(test[target], p_test),
       "brier": brier_score_loss(test[target], p_test),
       "precision": precision_score(test[target], pred_test, zero_division=0),
       "recall": recall_score(test[target], pred_test, zero_division=0)})

# A fila respeita limiar e capacidade: no máximo os 20 maiores riscos.
scored = test.assign(prob_atraso=p_test)
fila = scored[scored.prob_atraso >= threshold].sort_values("prob_atraso", ascending=False).head(20)
fila[["obra_id", "prob_atraso"]].to_csv("fila_semanal.csv", index=False)
joblib.dump({"model": pipe, "features": features, "threshold": threshold},
            "risco_atraso.joblib")
Depois do deploy: registre quem foi sinalizado, quem recebeu intervenção e qual foi o resultado. Sem separar efeito do risco e efeito da intervenção, o histórico futuro pode ensinar que obras tratadas “não eram arriscadas”.
Probabilidade precisa ser auditada: ROC-AUC mede ordenação, não calibração. Use curva de confiabilidade e Brier score; se necessário, calibre em dados separados antes de aplicar custos monetários ao limiar.

Aprendizado em lote (batch) x online

Lote ou atualização contínuavisual explicativo · animação leve

A frequência de treino deve acompanhar a velocidade de mudança do problema.

Uma dúvida comum de quem está começando: o modelo aprende sozinho, em tempo real, conforme os dados chegam? Depende de como você configura — existem dois jeitos bem diferentes de manter um modelo atualizado.

Em lote (batch) — o mais comum

O modelo é treinado de uma vez, com um monte de dados históricos acumulados, entra em produção, e só é atualizado de novo quando alguém roda o treino novamente com dados mais recentes — toda semana, todo mês, ou a cada nova obra concluída. Entre um retreino e outro, o modelo fica parado, usando o que aprendeu no último ciclo.

Online (incremental)

O modelo se ajusta sozinho, um pouco a cada novo dado que chega, sem esperar um ciclo de retreino completo. Faz sentido quando o padrão muda muito rápido ou o volume de dado novo é grande e constante o suficiente pra justificar isso.

Supermercado

Previsão de vendas — batch

Reprocessada toda semana com os dados da semana anterior. Não precisa reagir a cada venda individual, então batch é suficiente e mais simples de manter.

Construção civil

Estimativa de custo — batch

Os dados de obra (medições, avanço físico) não chegam em volume nem velocidade que justifiquem aprendizado online. Atualizar o modelo a cada obra concluída já é suficiente.

Varejo

Recomendação de produto — online

Grandes plataformas de e-commerce ajustam a recomendação em tempo real conforme você clica e navega, sem esperar um reprocessamento noturno.

Cuidado com online: como o modelo aprende sozinho e continuamente, um período de dados ruins (ou até manipulados de propósito) pode "contaminar" o modelo rapidamente, sem um humano revisando antes de valer pra produção.
Regra prática: comece sempre com batch — é mais simples de controlar e de depurar quando algo dá errado. Só migre pra online quando tiver comprovado que o ciclo de retreino periódico é rápido demais pra acompanhar a velocidade que o seu negócio muda.

Um modelo não é "pronto e esquecido". Depois que ele entra em uso real, o mundo continua mudando — e o modelo, não. Isso se chama drift (deriva): o modelo vai piorando aos poucos porque o padrão dos dados novos deixou de parecer com o padrão que ele aprendeu.

Exemplo rodável em Python

Pipeline reproduzívelvisual explicativo · animação leve

Pré-processamento, treino e avaliação devem ser encadeados para evitar vazamento.

Um microexemplo didático: prever se um cliente de supermercado vai cancelar o programa de fidelidade (churn), usando regressão logística. Ele demonstra a API; os projetos P1 e P2 acima mostram o processo completo com validação temporal, baseline, custo e artefato de produção.

Pra rodar: instale as bibliotecas com pip install pandas scikit-learn e execute o script num arquivo .py ou notebook Jupyter. O bloco abaixo cria dados fictícios apenas para demonstrar a sintaxe; dez linhas não permitem avaliar um modelo real. Em projetos reais, use amostra adequada, validação estratificada e um Pipeline para impedir vazamento entre treino e teste.
# 1. Importar bibliotecas
import pandas as pd
from sklearn.model_selection import train_test_split
from sklearn.linear_model import LogisticRegression
from sklearn.metrics import classification_report, confusion_matrix
from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler

# 2. Dados fictícios de clientes do supermercado
# (na vida real, isso viria do seu CRM/ERP)
dados = pd.DataFrame({
    "dias_sem_comprar": [5, 40, 12, 90, 3, 60, 8, 100, 15, 2],
    "reclamacoes":      [0, 3,  0, 4,  0, 2, 1, 5,   0, 0],
    "ticket_medio":     [120, 60, 200, 40, 300, 55, 150, 30, 180, 250],
    "cancelou":         [0, 1,  0, 1,  0, 1, 0, 1,   0, 0]   # 1 = cancelou
})

# 3. Separar features (X) do alvo (y)
X = dados[["dias_sem_comprar", "reclamacoes", "ticket_medio"]]
y = dados["cancelou"]

# 4. Separar treino e teste
X_treino, X_teste, y_treino, y_teste = train_test_split(
    X, y, test_size=0.3, random_state=42, stratify=y
)

# 5. Encadear transformação + modelo: o scaler aprende só no treino
modelo = Pipeline([
    ("scale", StandardScaler()),
    ("model", LogisticRegression(max_iter=1000))
])
modelo.fit(X_treino, y_treino)

# 6. Avaliar no que o modelo nunca viu
previsoes = modelo.predict(X_teste)
print(classification_report(y_teste, previsoes, zero_division=0))
print("Matriz de confusão:")
print(confusion_matrix(y_teste, previsoes))

# 7. Usar em um cliente novo
# (30 dias sem comprar, 1 reclamação, ticket médio R$90)
cliente_novo = pd.DataFrame({
    "dias_sem_comprar": [30],
    "reclamacoes": [1],
    "ticket_medio": [90]
})
probabilidade = modelo.predict_proba(cliente_novo)[0][1]
print(f"Probabilidade de cancelar: {probabilidade:.0%}")

O que trocar pra usar em outro setor

  • Construção civil (risco de atraso): troque as colunas por m2_obra, num_fornecedores, prazo_dias e o alvo por atrasou.
  • Varejo (devolução de produto): troque por categoria_produto, preco, canal_venda e o alvo por devolveu.
  • Quer trocar de modelo? a interface fit/predict costuma permanecer, mas o pré-processamento, a calibração, os hiperparâmetros e até a métrica podem mudar. Árvores não exigem escala; categorias exigem codificação; probabilidades usadas em decisões podem exigir calibração.

Glossário

Mapa mentalvisual explicativo · animação leve

Os termos se conectam em quatro grupos: dados, treino, avaliação e produção.

  • Feature: cada variável de entrada usada pelo modelo (m², dia da semana, categoria).
  • Target / rótulo: a resposta que o modelo tenta prever.
  • Treino / teste: conjunto de dados usado para aprender vs. usado para avaliar.
  • Overfitting (sobre-ajuste): modelo decorou o treino, erra em dado novo.
  • Underfitting (sub-ajuste): modelo simples demais, erra em tudo.
  • Regularização: técnica que penaliza a complexidade do modelo pra evitar sobre-ajuste.
  • Hiperparâmetro: configuração do modelo escolhida por você antes do treino (ex: número de árvores no Random Forest, valor de K no K-means).
  • Baseline: um modelo simples de referência (ex: "prever sempre a média") pra saber se o modelo complexo realmente vale a pena.
  • Pipeline: a sequência automatizada de passos, de dado bruto até predição.
  • Inferência: usar o modelo já treinado pra prever um caso novo, no dia a dia.
  • Bagging: técnica do Random Forest de treinar várias árvores em amostras aleatórias diferentes e juntar os votos.
  • Boosting: ensemble onde cada modelo novo corrige o erro do anterior (ex: XGBoost).
  • Lazy learner: modelo (como o KNN) que não constrói fórmula no treino, só guarda os dados e calcula tudo na hora da previsão.
  • Feature engineering: processo de criar, transformar e escolher as melhores variáveis de entrada pro modelo.
  • One-hot encoding: transformar uma variável categórica (texto) em várias colunas numéricas de 0 e 1.
  • Normalização/padronização: colocar variáveis em escalas parecidas antes de treinar modelos sensíveis a escala.
  • Viés (bias): quando o modelo reproduz ou amplifica uma discriminação já presente nos dados históricos.
  • Função de custo: régua de erro usada durante o treino pra guiar o próprio modelo.
  • Função de avaliação: métrica usada depois do treino pra você julgar se o modelo está bom o suficiente pro negócio.
  • Aprendizado em lote (batch): modelo retreinado periodicamente (semana, mês) em vez de se atualizar sozinho a cada dado novo.
  • Aprendizado online: modelo que se ajusta sozinho, aos poucos, a cada novo dado que chega.
  • Sazonalidade: padrão que se repete em ciclo fixo (semana, mês, ano) numa série temporal.
  • Drift: quando o modelo em produção piora porque o padrão do mundo real mudou.
  • SHAP / feature importance: técnicas pra explicar por que o modelo decidiu o que decidiu.
  • Teste A/B: comparação controlada entre "com modelo" e "sem modelo" pra provar resultado real.
  • Classes desbalanceadas: quando uma categoria (ex: fraude) é muito mais rara que a outra nos dados.
  • K-fold (validação cruzada): técnica de avaliar o modelo treinando e testando várias vezes com pedaços diferentes dos dados, em vez de uma única divisão treino/teste.
  • Validação: partição usada para escolher modelo, hiperparâmetros e limiar; é diferente do teste final.
  • Vazamento de alvo (target leakage): uso de informação indisponível no instante real da previsão.
  • Calibração: propriedade em que previsões de 70% acontecem aproximadamente 70% das vezes.
  • Limiar: ponto que converte probabilidade em ação; deve refletir custos e capacidade operacional.
  • WAPE: soma dos erros absolutos dividida pela soma dos valores reais; útil para demanda agregada, mas deve ser segmentada.
  • Data drift: mudança na distribuição das entradas. Não prova sozinho que o modelo perdeu qualidade.
  • Concept drift: mudança na relação entre entradas e alvo; exige observar resultados reais, que podem chegar com atraso.
Sobre os selos de profundidade: a partir de agora, quando você trouxer um conceito novo que estudou em outro lugar, eu marco com um selo — essencial (vale aprender bem, você vai usar sempre), bom saber (ajuda, mas dá pra aprender só o suficiente pra reconhecer quando aparecer) ou avançado / opcional (interessante, mas pode empurrar pra depois sem prejuízo agora). Isso evita que o caderno vire uma lista de tudo que existe em ML, sem noção do que realmente importa aplicar primeiro.

Modelo · regressão

Regressão linear

Linha e resíduosvisual explicativo · animação leve

A linha representa tendência; distâncias até os pontos representam erros.

Serve pra prever um número contínuo (preço, quantidade, custo, tempo). A ideia é achar a reta (ou plano, com várias variáveis) que melhor passa no meio dos pontos dos dados.

vendas = a + b1·(promoção) + b2·(dia_da_semana) + b3·(temperatura)

O modelo "aprende" os valores de a, b1, b2, b3 — o quanto cada variável pesa no resultado final.

Supermercado

Previsão de vendas do mês

Entrada: histórico de promoções, sazonalidade, número de clientes. Saída: faturamento previsto.

Construção civil

Custo por m²

Entrada: m² construído, padrão de acabamento, região. Saída: custo total estimado da obra.

Varejo

Previsão de demanda

Entrada: preço, concorrência, época do ano. Saída: quantas unidades vão ser vendidas.

Quando usar: sempre que a pergunta for "quanto?" — quanto vai vender, quanto vai custar, quanto tempo vai levar.
Desafios na implementação
  • Relação não linear: economia de escala numa obra grande (custo por m² cai conforme a obra cresce) não é uma reta. Como contornar: transformar a variável (log, raiz, polinômio) ou migrar pra um modelo que capta curvas sozinho, como boosting (folha 11).
  • Sensibilidade a outliers: uma obra atípica gigante no meio de obras residenciais distorce a reta toda. Como contornar: tratar/remover outliers antes de treinar (folha 22).
  • Multicolinearidade: quando duas variáveis de entrada são muito parecidas entre si (m² construído e m² do terreno), o modelo "confunde" o peso de cada uma. Como contornar: remover uma das variáveis redundantes, ou usar regularização (Ridge/Lasso).

Regressão logística

Probabilidade e limiarvisual explicativo · animação leve

A curva produz probabilidade; um limiar transforma a saída em classe.

Apesar do nome, não é regressão — é o modelo de classificação mais básico. Em vez de prever um número qualquer, ela prevê uma probabilidade entre 0 e 1 de algo acontecer (sim ou não).

probabilidade(cliente cancela) = sigmoid( a + b1·(dias sem comprar) + b2·(reclamações) )

A função sigmoid só existe pra "espremer" qualquer número dentro da faixa 0–1, virando uma probabilidade. Se a probabilidade passar de um corte (ex: 0,5), você classifica como "sim".

Varejo

Cliente vai comprar a oferta

Probabilidade de um cliente responder a uma promoção enviada por e-mail/SMS, com base no histórico dele.

Regressão linear vs logística: linear prevê "quanto" (um número); logística prevê "qual a chance de sim" (uma probabilidade que vira sim/não).
Desafios na implementação
  • Fronteira simples demais: assume uma separação relativamente linear e não capta bem interações complexas entre variáveis sozinha. Como contornar: criar features de interação manualmente (folha 22), ou migrar pra árvore/boosting se a relação for muito mais complexa.
  • Classe rara: com poucos exemplos da categoria que importa (fraude, cancelamento), o modelo fica instável. Como contornar: técnicas de classes desbalanceadas (folha 19).

Perguntas que dividem os dadosvisual explicativo · animação leve

Cada nó escolhe uma regra; cada folha entrega uma previsão.

Uma sequência de perguntas do tipo "se/então", igual um fluxograma. É o modelo mais fácil de explicar pra qualquer pessoa do negócio, porque dá pra desenhar as regras exatas que ele usou.

Como ela escolhe as perguntas

A cada divisão (nó), o algoritmo testa várias perguntas possíveis ("m² maior que 500?", "mais de 3 fornecedores?") e escolhe a que separa melhor as categorias. Essa "qualidade da pergunta" é medida por um índice chamado Gini ou entropia — na prática, quanto mais "pura" (mais concentrada numa única categoria) fica cada ramo depois da pergunta, melhor ela foi.

Ex. construção civil: "Se a obra tem mais de 500m² e o prazo é menor que 6 meses e tem mais de 3 fornecedores envolvidos → risco alto de atraso."

O hiperparâmetro que mais importa: profundidade máxima

É quantas perguntas seguidas a árvore pode fazer antes de dar a resposta final.

  • Árvore muito profunda: cria uma regra específica demais pra cada caso raro do treino — decora em vez de aprender o padrão (sobre-ajuste, folha 16).
  • Árvore rasa demais: só consegue fazer 1 ou 2 perguntas, e erra por ser simples demais (sub-ajuste, folha 16).
Supermercado

Reposição urgente

"Se estoque atual menor que 3 dias de venda média e produto é perecível → repor hoje." Uma regra simples que o time do depósito consegue aplicar sem abrir sistema nenhum.

Quando usar sozinha: quando a transparência da regra importa mais do que a precisão máxima — por exemplo, em decisões que precisam ser explicadas a um auditor, a um cliente ou a um órgão regulador.
Desafios na implementação
  • Instabilidade: uma pequena mudança nos dados de treino pode gerar uma árvore de regras completamente diferente. Como contornar: não confiar numa árvore isolada pra decisão de alto risco — prefira Random Forest ou boosting (folhas 08 e 11).
  • Tendência a decorar: cresce demais e vira uma lista de exceções memorizadas. Como contornar: limitar a profundidade (max_depth) e podar (pruning) usando validação cruzada.

Modelo · classificação

Random Forest

Muitas árvores, uma decisãovisual explicativo · animação leve

A diversidade entre árvores reduz a fragilidade de uma árvore isolada.

Em vez de treinar uma árvore só, treina centenas de árvores diferentes e junta o "voto" de todas elas — a categoria mais votada vence. Essa técnica de combinar vários modelos se chama bagging.

Por que treinar centenas ao invés de uma só

Cada árvore da floresta vê apenas uma amostra aleatória dos dados (sorteada com reposição) e, a cada divisão, só pode escolher entre um subconjunto aleatório das variáveis. Isso faz as árvores "discordarem" entre si de propósito. Quando você tira a média/maioria de várias opiniões diferentes, os erros individuais de cada árvore se cancelam, e o resultado final fica muito mais estável do que confiar numa árvore só.

Hiperparâmetros principais: número de árvores (mais árvores = mais estável, até um ponto de rendimento decrescente), profundidade máxima de cada árvore, e quantas variáveis sortear a cada divisão.
Varejo

Previsão de devolução

Combina o "voto" de 300 árvores treinadas com categoria do produto, canal de venda e histórico do cliente pra prever se um item será devolvido.

Supermercado

Ruptura de estoque

Combina sazonalidade, estoque atual, tempo de reposição do fornecedor e histórico de vendas — variáveis demais pra uma árvore só lidar bem sozinha.

Trade-off: ganha muita precisão, mas perde a transparência de "olhar a regra" que a árvore única tinha — vira uma decisão de centenas de árvores somadas. Pra recuperar um pouco de explicação, existe a técnica de feature importance (detalhada na folha 31, explicabilidade).
Desafios na implementação
  • Custo computacional: fica pesado com muitas árvores e bases muito grandes — treino e previsão mais lentos. Como contornar: reduzir número de árvores/profundidade, ou migrar pra boosting (folha 11), que costuma precisar de menos árvores pra mesma precisão.
  • Perda de transparência: já não dá pra "ler a regra" como numa árvore só. Como contornar: usar feature importance (folha 31) pra pelo menos entender o peso de cada variável na decisão.

KNN (K-vizinhos mais próximos)

Decisão pela vizinhançavisual explicativo · animação leve

Um caso novo é comparado aos casos mais próximos; escala e K mudam o resultado.

Pra classificar um caso novo, o KNN mede a distância dele até todos os casos que já existem no histórico, pega os K mais parecidos (os "vizinhos"), e copia a resposta da maioria deles.

Um jeito de pensar nisso que você já conhece

Isso é exatamente o método comparativo de mercado usado na avaliação de imóveis e obras: pra estimar o valor de um imóvel, você compara com imóveis parecidos (mesma região, m² similar) que já foram vendidos recentemente, e usa o preço deles como referência. O KNN faz matematicamente a mesma coisa, só que em escala e com qualquer tipo de dado.

Detalhes que fazem diferença

  • É um "lazy learner": não constrói fórmula nenhuma durante o treino — só guarda os dados. Isso deixa o "treino" instantâneo, mas a previsão fica mais lenta quando a base é enorme, porque ele recalcula a distância até tudo, toda vez.
  • Escolha de K: normalmente um número ímpar (evita empate de votos), testado com validação cruzada até achar o valor que erra menos.
  • Sensível à escala das variáveis: se uma variável vai de 40 a 2000 (m² da obra) e outra vai de 1 a 5 (número de banheiros), a primeira domina o cálculo de distância sem querer — por isso normalizar os dados antes é obrigatório aqui (folha 23, normalização).
Supermercado

Sucesso de produto novo

Pra decidir se um produto novo vai ser "sucesso" ou "encalhe", olha os 10 produtos mais parecidos (mesma categoria, preço similar) e vê o que aconteceu com eles.

Desafios na implementação
  • Fica lento com muitos dados: recalcula distância pra tudo a cada previsão nova. Como contornar: usar estruturas de indexação (KD-tree/Ball-tree) ou reduzir a base com amostragem.
  • "Maldição da dimensionalidade": com muitas variáveis, a noção de "distância" perde sentido — tudo fica parecido demais entre si. Como contornar: reduzir dimensões (PCA, folha 25) ou selecionar só as features mais relevantes antes (folha 22).

SVM (Support Vector Machine)

Separar com margemvisual explicativo · animação leve

A fronteira procura ficar o mais distante possível dos pontos críticos.

Entre duas categorias, o SVM procura a linha (ou plano) de separação que deixa a maior margem de segurança possível em relação aos exemplos mais próximos de cada lado — esses exemplos "de fronteira" são chamados de vetores de suporte, o que dá nome ao modelo.

E quando os dados não são separáveis por uma linha reta?

O SVM usa um truque chamado kernel: ele projeta os dados numa dimensão a mais até que uma separação reta passe a fazer sentido nesse novo espaço. Você não precisa entender a matemática por trás — só saber que essa opção existe (o kernel "RBF" é o mais usado quando a separação não é óbvia).

Hiperparâmetro C: controla o quanto o modelo tolera alguns erros de classificação em troca de uma margem mais larga (e mais generalizável). C baixo tolera mais erro e generaliza melhor; C alto tenta acertar tudo no treino e arrisca decorar demais.
Construção civil

Controle de qualidade de material

Separar material "aprovado" de "reprovado" no controle de qualidade, com base em resistência e densidade medidas em laboratório.

Varejo

Linha premium x econômica

Classificar um produto novo como linha premium ou linha econômica com base em preço, margem e características técnicas.

Desafios na implementação
  • Escolha de kernel e hiperparâmetros: não existe uma resposta óbvia de antemão pra C e pro tipo de kernel — exige tentativa e erro. Como contornar: busca sistemática (grid search) combinada com validação cruzada.
  • Fica lento em bases grandes: o custo computacional cresce rápido com o número de exemplos. Como contornar: usar uma amostra representativa menor, ou trocar por um modelo que escale melhor (boosting, folha 11).

Modelo · ensemble

Boosting (XGBoost / LightGBM)

Correções sequenciaisvisual explicativo · animação leve

Cada modelo novo concentra-se nos erros que o conjunto anterior ainda não resolveu.

Hoje em dia, na indústria, o Random Forest perdeu espaço pro boosting na maioria dos projetos de dados em tabela (planilha, banco de dados). É praticamente o padrão de mercado.

A diferença pro Random Forest, sem fórmula

Random Forest treina centenas de árvores independentes e tira a média. Boosting treina as árvores uma depois da outra, e cada árvore nova foca em corrigir os erros que a árvore anterior deixou passar.

Analogia: Random Forest é como perguntar a opinião de 300 mestres de obra diferentes e tirar a média. Boosting é como ter um mestre de obra que revisa o trabalho do anterior e corrige só o que ficou errado, e repete isso centenas de vezes.
Supermercado

Previsão de demanda por SKU

Milhares de produtos, cada um com sazonalidade diferente — boosting costuma vencer regressão linear e Random Forest nesse tipo de problema.

Construção civil

Estimativa de custo mais precisa

Quando você já tem centenas de obras históricas, boosting geralmente reduz o erro de estimativa em relação a um Random Forest simples.

Desafios na implementação
  • Overfita fácil se não regularizado: por ser mais poderoso, exige mais atenção. Como contornar: early stopping (parar de adicionar árvores quando o erro de validação para de cair), limitar profundidade e taxa de aprendizado.
  • Treino sequencial mais lento: cada árvore depende da anterior, o que dificulta paralelizar como no Random Forest. Como contornar: bibliotecas como LightGBM já são otimizadas pra isso; reduzir número de rodadas com early stopping também ajuda.

K-means e clustering

Centroides e gruposvisual explicativo · animação leve

Atribuir pontos e recalcular centros se repete até estabilizar.

O K-means separa os dados em K grupos (você escolhe quantos) de forma que os itens dentro de cada grupo sejam o mais parecidos possível entre si, e o mais diferentes possível dos outros grupos.

Como ele funciona, sem fórmula

  1. Você escolhe quantos grupos quer (ex: 3 tipos de cliente).
  2. O algoritmo joga 3 "centros" aleatórios no meio dos dados.
  3. Cada cliente é colocado no grupo do centro mais próximo dele.
  4. Cada centro se move pro meio do seu grupo.
  5. Repete os passos 3 e 4 até os grupos pararem de mudar.
Supermercado

3 perfis de cliente

K-means com ticket médio e frequência de compra separa em "compra pouco e raro", "compra muito e raro (estoca)" e "compra pouco e sempre (fidelizado)".

Construção civil

Perfis de fornecedor

Agrupar fornecedores por preço médio e prazo de entrega, pra saber quem negociar em cada tipo de obra (urgente vs. sem pressa de prazo).

Como escolher o K (quantos grupos)? Na prática, você testa vários valores de K e olha onde o ganho de "separar mais" para de compensar — isso se chama método do cotovelo (elbow method).

Outras variações que você vai ouvir falar: clustering hierárquico (monta uma árvore de grupos dentro de grupos) e DBSCAN (agrupa por densidade, bom pra achar pontos fora do padrão — útil pra detectar fraude ou anomalia).

Desafios na implementação
  • Escolher o K de antemão: o resultado muda bastante dependendo do K escolhido. Como contornar: método do cotovelo ou coeficiente de silhueta pra escolher com mais critério.
  • Assume grupos "redondos" e de tamanho parecido: se os grupos reais tiverem formato irregular, o K-means força uma divisão que não faz sentido. Como contornar: usar DBSCAN ou clustering hierárquico quando os grupos não parecerem esféricos.
  • Sensível à escala e a outliers: Como contornar: normalizar antes (folha 23) e tratar outliers.

Tempo e negócio

Séries temporais

Tendência e sazonalidadevisual explicativo · animação leve

A ordem temporal importa; passado e futuro não podem ser embaralhados livremente.

Vendas de supermercado e prazos de obra têm uma característica que a regressão linear comum ignora: o valor de hoje depende do valor de ontem, e existe repetição em ciclos (semana, mês, ano). Tratar isso como "só mais uma variável" perde muita precisão.

Três componentes que você precisa reconhecer

  • Tendência: o negócio está crescendo, caindo ou estável ao longo dos anos.
  • Sazonalidade: padrão que se repete em ciclo fixo — Natal, dia de pagamento (dia 5 e 20), verão. Previsível e recorrente.
  • Ruído: variação aleatória que nenhum modelo vai conseguir prever (uma chuva forte inesperada, um evento local pontual).
Supermercado

Previsão de vendas diária

O modelo precisa saber que dia 5 e 20 (pagamento de salário) vendem mais, dezembro vende muito mais que fevereiro, e segunda-feira vende menos que sábado.

Construção civil

Previsão de cronograma

Chuva no período das águas atrasa concretagem todo ano — isso é sazonalidade, não "azar", e dá pra planejar o cronograma considerando esse padrão histórico.

Modelos comuns pra isso: ARIMA (estatístico, bom pra séries mais simples e curtas) e Prophet (do Facebook/Meta, feito pra ser fácil de usar com sazonalidade e feriados já embutidos). Pra séries mais complexas, com muitas variáveis externas, boosting e redes neurais (folha 26) também entram em jogo.

Regra prática: antes de escolher modelo de série temporal, separe visualmente tendência, sazonalidade e ruído num gráfico. Se você não sabe qual sazonalidade existe no seu negócio, o modelo também não vai adivinhar sozinho — sazonalidade forte demais numa base de dados curta (menos de 2 ciclos completos) não dá pra aprender direito.
Desafios na implementação
  • Dados faltantes quebram o padrão: um dia sem registro de vendas confunde a sazonalidade. Como contornar: preencher gaps com interpolação ou com a média do mesmo dia em ciclos anteriores.
  • Mudanças estruturais: abertura de concorrente, mudança de escopo de obra ou eventos atípicos invalidam o padrão histórico de uma hora pra outra. Como contornar: marcar esses eventos como uma variável externa no modelo, ou re-treinar assim que a mudança for identificada.

Como o modelo aprende

Função de custo (erro)

Erro transformado em númerovisual explicativo · animação leve

A função de custo define o que significa errar durante o treinamento.

Todo modelo, no fundo, faz a mesma coisa: chuta uma resposta, mede o quão errado chutou, e ajusta pra errar menos da próxima vez. A "função de custo" é a régua que mede esse erro durante o treino, pra guiar o próprio modelo.

MSE — erro quadrático médio (regressão)

MSE = média( (valor_real - valor_previsto)² )

Eleva ao quadrado pra dar peso maior aos erros grandes e ignorar o sinal (positivo/negativo).

Ex.: se o modelo previu R$ 50.000 de custo de obra e o real foi R$ 55.000, o erro daquele caso é (55.000 - 50.000)² = 25.000.000. O MSE é a média disso em todas as obras testadas.

Entropia cruzada (classificação)

Usada quando a saída é uma probabilidade (regressão logística, redes neurais de classificação). Pune muito mais forte quando o modelo está confiante e errado do que quando está em dúvida e erra.

Ex.: o modelo dizer "98% de certeza que o cliente não vai cancelar" e o cliente cancelar é punido muito mais do que dizer "55% de chance" e errar.

Você não precisa calcular essas fórmulas na mão — bibliotecas prontas fazem isso. O que importa é entender o que está sendo minimizado — e por que isso não é a mesma coisa que "a nota final do modelo" (veja a folha 17).

Como o modelo aprende

Gradiente descendente

Descendo a superfície de errovisual explicativo · animação leve

A taxa de aprendizado controla o tamanho dos passos rumo a parâmetros melhores.

É o método que o modelo usa pra diminuir o erro aos poucos, ajustando seus parâmetros em pequenos passos, sempre na direção que reduz o erro.

Analogia da obra: imagine que você está tentando prever o custo por m² e a cada obra terminada você compara sua estimativa com o custo real. Você não acerta o peso ideal de cada variável de uma vez — você vai ajustando o "peso" que dá pro tamanho da obra, pro tipo de acabamento, etc., um pouquinho a cada obra nova, sempre na direção que reduz o erro da estimativa. Isso, repetido milhares de vezes de forma automática, é o gradiente descendente.

Um detalhe que você vai ouvir bastante: a taxa de aprendizado

É o "tamanho do passo" que o modelo dá a cada ajuste.

  • Passo grande demais: o modelo "pula" a melhor solução e nunca estabiliza.
  • Passo pequeno demais: o modelo demora uma eternidade pra aprender.

Na prática, ferramentas prontas (scikit-learn, XGBoost etc.) já escolhem valores razoáveis por padrão — você só precisa saber que esse parâmetro existe e pode ser ajustado se o treino não estiver convergindo.

Sobre-ajuste e sub-ajuste

Capacidade e generalizaçãovisual explicativo · animação leve

O objetivo é capturar o padrão sem memorizar o ruído.

Overfitting (sobre-ajuste): o modelo "decorou" os dados de treino, incluindo o ruído e as exceções, em vez de aprender o padrão geral. Fica ótimo nos dados que já viu e ruim em dados novos.

Ex. supermercado: um modelo de previsão de vendas que "decorou" que toda terça-feira de chuva no bairro X vendeu 40 pães — em vez de aprender que chuva reduz vendas em geral, ele criou uma regra específica demais que não se repete.

Underfitting (sub-ajuste): o oposto — o modelo é simples demais pra captar o padrão real, e erra tanto no treino quanto em dados novos.

Ex. construção civil: prever o custo de qualquer obra usando só "preço médio por m²" fixo, ignorando padrão de acabamento e região — erra tanto pra obra simples quanto pra obra de luxo.

Como saber qual dos dois está acontecendo

  • Erro baixo no treino, erro alto no teste → sobre-ajuste (overfitting).
  • Erro alto no treino e no teste → sub-ajuste (underfitting).
  • Erro baixo nos dois → o modelo está no ponto certo.

Como evitar sobre-ajuste (overfitting)

  • Regularização (L1/Lasso, L2/Ridge): penaliza pesos grandes no modelo, forçando ele a ficar mais "simples" mesmo que isso signifique errar um pouco mais no treino.
  • Mais dados de treino: quanto mais exemplos variados, mais difícil o modelo decorar em vez de aprender o padrão real.
  • Simplificar o modelo: duas frentes diferentes — (1) escolher um algoritmo mais simples, trocando um modelo complexo (boosting, rede neural) por um mais enxuto (regressão linear/ logística, árvore rasa); e (2) reduzir o número de parâmetros do modelo que você já está usando (menos variáveis de entrada, menos árvores no Random Forest, menos rodadas no boosting).
  • Early stopping: parar o treino assim que o erro de validação parar de melhorar, mesmo que o erro de treino continue caindo.
  • Validação cruzada: ajuda a perceber o sobre-ajuste cedo, antes de ir pro dado de teste final.
Ex. construção civil: com só 30 obras no histórico, um boosting com muitas árvores tende a decorar. Às vezes a solução não é ajustar hiperparâmetro nenhum — é simplesmente trocar por uma regressão linear, que generaliza melhor justamente por ser mais simples com pouco dado disponível.

Como evitar sub-ajuste (underfitting)

  • Modelo mais complexo: trocar regressão linear por boosting, aumentar a profundidade da árvore.
  • Mais features relevantes: feature engineering (folha 22) — o modelo pode estar simples porque falta informação, não porque ele é fraco.
  • Reduzir a regularização: se ela estiver penalizando o modelo com força demais.
  • Treinar por mais tempo: em modelos baseados em gradiente descendente (folha 15), às vezes o treino parou cedo demais.

A base de tudo: dividir os dados em treino, validação e teste

  • Treino (≈70%): dados que o modelo usa pra aprender.
  • Validação (≈15%): usada pra ajustar as configurações do modelo durante o desenvolvimento.
  • Teste (≈15%): guardada intocada, só usada no final, pra saber a nota real do modelo.

Validação cruzada k-fold, na prática

Em vez de separar treino/teste uma única vez, você divide os dados em K pedaços (folds) — normalmente 5 ou 10. O modelo treina em K-1 pedaços e testa no pedaço restante, repetindo isso K vezes até que cada pedaço tenha sido o "teste" uma vez. A nota final é a média do erro nas K rodadas.

Construção civil

Onde isso importa de verdade

Com só 30 obras concluídas, uma única divisão 70/30 deixa o teste com 9 obras — se por azar essas 9 forem atípicas, sua avaliação do modelo fica distorcida. Com k-fold (K=5), cada grupo de 6 obras vira teste por vez, e a média das 5 rodadas dá uma visão bem mais realista.

Supermercado

Onde importa menos

Com 2 anos de vendas diárias (730 linhas), uma única divisão já é razoavelmente estável — o volume de dados absorve boa parte do "azar" de um recorte ruim. K-fold ainda ajuda, mas o ganho é menor.

essencial — vale aprender bem

Isso é diferente de casos como SHAP ou drift, que são bons de saber mas dá pra empurrar pra depois. K-fold é usado o tempo todo, em praticamente qualquer projeto sério, e é simples de aplicar — no scikit-learn é uma função só (cross_val_score).

Até onde vale ir

  • Precisa saber: o que é, por que existe (nota mais confiável que um único split, principalmente com poucos dados) e o procedimento geral (divide em K, roda K vezes, tira a média).
  • Bom saber, no nível de uso: que existe uma variação chamada k-fold estratificado, que garante que a proporção de cada categoria fique parecida em todos os folds — útil quando você tem classes desbalanceadas (folha 19), como fraude ou atraso grave.
  • Não precisa ir atrás: a demonstração estatística de por que k-fold reduz viés e variância da estimativa, nem variações mais acadêmicas como leave-one-out ou validação cruzada aninhada (nested cross-validation). Isso é papo de quem pesquisa método, não de quem aplica.
Regra prática de uso: k-fold serve pra avaliar e ajustar o modelo durante o desenvolvimento (inclusive pra escolher hiperparâmetros). Ele não substitui guardar um conjunto de teste final intocado (a divisão treino/validação/teste que você viu acima, nesta mesma folha) — os dois trabalham juntos, não um no lugar do outro.

Analogia: a função de custo é o termômetro interno que o modelo usa sozinho, a cada ajuste, pra saber se está esquentando ou esfriando o erro. A função de avaliação é o boletim final que você, humano, usa pra aprovar ou reprovar o modelo pronto.

Por que às vezes elas discordam

Você pode treinar um modelo de fraude minimizando entropia cruzada (função de custo) durante o treino, mas na hora de avaliar se ele "vale a pena", olhar pro F1-score ou até direto pra quanto dinheiro a empresa deixou de perder (função de avaliação de negócio) — que é o que realmente importa, mesmo que essa métrica de dinheiro não sirva matematicamente pra ajustar o modelo durante o treino (não é "suave" o bastante pro gradiente descendente calcular em cima dela).

Fique atento: as duas normalmente apontam na mesma direção (menor custo no treino costuma vir com métrica melhor), mas nem sempre. Nunca aprove um modelo só porque o erro de treino caiu — sempre confira a métrica de avaliação que importa pro seu negócio (folha 18) em dados que o modelo nunca viu.

Métricas de avaliação

Matriz de confusãovisual explicativo · animação leve

Os quatro resultados revelam tipos de erro escondidos pela acurácia.

Para regressão (previsão de número)

  • MAE (erro médio absoluto): "em média, erro R$ X pra mais ou pra menos". Fácil de explicar pro gestor de obra.
  • RMSE: parecido com o MAE, mas pune mais forte os erros grandes.
  • R²: de 0 a 1, mostra o quanto o modelo explica da variação real (1 = perfeito).

Para classificação (sim/não, categoria)

Aqui a métrica errada engana. Se 95% dos clientes de um supermercado não cometem fraude no cartão fidelidade, um modelo "burro" que sempre diz "não é fraude" já acerta 95% — e é inútil.

  • Acurácia: % total de acertos. Só é confiável quando as categorias são equilibradas.
  • Precisão: das vezes que o modelo disse "é fraude", quantas realmente eram.
  • Recall (sensibilidade): das fraudes que realmente aconteceram, quantas o modelo pegou.
  • F1-score: equilíbrio entre precisão e recall, num número só.
Ex. varejo — prevenção de perdas: se seu modelo prioriza recall alto, ele pega quase todas as fraudes, mas também aponta gente inocente como suspeita (mais falsos alarmes). Se prioriza precisão alta, quando ele aponta fraude quase sempre é fraude de verdade, mas deixa passar mais casos. A escolha depende do custo de cada erro no seu negócio.

Como o modelo aprende

Classes desbalanceadas

Classe rara, impacto grandevisual explicativo · animação leve

95% de acurácia pode ser inútil quando os 5% raros são o que realmente importa.

Fraude, atraso grave de obra e cancelamento de cliente têm algo em comum: são raros (às vezes 1 a 5% dos casos). Um modelo treinado direto nesses dados tende a "aprender" que a resposta mais segura é sempre dizer "não aconteceu" — e erra justo nos casos que mais importam.

Técnicas pra lidar com isso

  • Oversampling (SMOTE): criar exemplos sintéticos da classe rara pra equilibrar o treino.
  • Undersampling: reduzir a quantidade de exemplos da classe comum, pra ficar mais equilibrado com a rara.
  • Peso de classe (class weight): dizer pro modelo "errar na classe rara custa 10x mais caro que errar na classe comum" — a maioria das bibliotecas tem essa opção pronta.
Varejo

Fraude no cartão fidelidade

De 100 mil transações, 300 são fraude (0,3%). Sem tratar o desbalanceamento, o modelo aprende a nunca marcar fraude.

O erro mais caro

Correlação não é causalidade

Associação não prova causavisual explicativo · animação leve

Uma terceira variável pode explicar por que duas grandezas se movem juntas.

O modelo encontra padrões que acontecem juntos. Isso não quer dizer que um causa o outro. Confundir os dois leva a decisões de negócio erradas — o modelo não vai te avisar disso, é você quem precisa checar.

Ex. clássico: um supermercado percebe que dias com mais vendas de sorvete também têm mais afogamentos registrados na cidade. Sorvete não causa afogamento — os dois sobem juntos por causa de um terceiro fator: calor. Tomar decisão em cima da correlação errada (ex: "vamos parar de vender sorvete pra reduzir acidentes") seria um absurdo, mas em dados de negócio esse tipo de erro é mais sutil e mais fácil de cair.

Não é a reunião que causa atraso — obras problemáticas é que geram mais reuniões de emergência. A causa é o problema, a reunião é consequência, não o contrário.

O erro mais caro

Viés em modelos (bias)

Onde o viés entravisual explicativo · animação leve

Coleta, rótulo, objetivo, limiar e uso podem produzir impactos injustos.

Um modelo aprende exatamente o que está nos dados — inclusive os preconceitos e desigualdades que já existiam antes dele. Isso se chama viés algorítmico, e ele não aparece como um "erro" técnico óbvio: o modelo pode ter métricas ótimas no geral (folha 18) e, mesmo assim, estar discriminando um grupo específico.

Três tipos de viés mais comuns

  • Viés histórico: os dados refletem decisões humanas passadas que já eram enviesadas (ex: um histórico de contratação que sempre preferiu um perfil específico de candidato).
  • Viés de proxy: uma variável "neutra" na aparência carrega, na prática, informação sensível por tabela — CEP correlacionado com renda, por exemplo.
  • Viés de amostragem: os dados de treino não representam bem toda a população real (ex: câmeras de segurança concentradas só em certas lojas ou bairros).
Varejo

Crédito com CEP como variável

Um modelo de aprovação de crediário que usa CEP como variável pode acabar penalizando bairros inteiros de forma indireta, mesmo sem "querer" discriminar por classe social.

Construção civil

Um modelo treinado no histórico de contratações da empresa pode reproduzir um viés de gênero ou idade que já existia nas contratações passadas, sem que ninguém tenha programado isso de propósito.

Como mitigar

  • Auditar os dados de treino antes de confiar neles — pergunte "esse histórico já era justo?"
  • Evitar ou tratar variáveis proxy: teste se remover a variável muda o resultado pra diferentes grupos.
  • Medir a métrica de avaliação separadamente por subgrupo (não só no geral) — um modelo pode ter 90% de acurácia geral e mesmo assim errar muito mais pra um grupo específico.
  • Manter humano na decisão final em casos de alto impacto (crédito, contratação, saúde) — o modelo sugere, a pessoa decide.
Viés não se resolve "uma vez só": precisa ser monitorado junto com o drift (folha 29), porque o padrão da sociedade — e da própria empresa — também muda com o tempo.

Feature engineering — como achar as melhores variáveis

Dado bruto vira sinal útilvisual explicativo · animação leve

Boas features revelam padrões sem usar informação que só existirá no futuro.

Na prática, 70–80% do trabalho de um projeto de ML é preparar dado, não escolher o modelo. Feature engineering é o nome do processo de transformar dado bruto em variáveis que realmente ajudam o modelo a aprender o padrão — criar, transformar, combinar e escolher quais colunas entram no treino.

Como determinar quais são as melhores features

  • Correlação com o alvo: veja se a variável realmente se move junto com o que você quer prever (lembrando da armadilha da folha 20 — correlação não garante causa, mas ainda ajuda a triagem inicial).
  • Feature importance de modelos como Random Forest/boosting: treine um modelo rápido e veja quais variáveis ele usou mais (liga com explicabilidade, folha 31).
  • Remover e comparar (ablation): tire uma variável de cada vez e veja se a métrica de avaliação (folha 18) piora — se quase não muda, ela provavelmente não ajudava.
  • Conhecimento de negócio: muitas vezes a melhor feature não vem de estatística nenhuma — vem de quem conhece o processo. O mestre de obra já sabe que "tipo de fundação" importa mais pro custo do que "cor da fachada", antes mesmo de rodar qualquer modelo.
Mais features nem sempre é melhor: variáveis irrelevantes ou redundantes atrapalham — voltam como ruído e contribuem pro sobre-ajuste (folha 16). Prefira poucas features fortes a muitas fracas.

Tratamento de dados faltantes

Preencher com média/mediana, ou criar uma categoria "não informado" — nunca deixar em branco, porque a maioria dos modelos simplesmente quebra ou distorce o resultado com valores vazios.

Normalização e one-hot encoding — as duas técnicas de pré-processamento mais usadas no dia a dia — ganharam folhas próprias: veja a folha 23 e a folha 24.

Normalização (e padronização)

Escalas comparáveisvisual explicativo · animação leve

Distâncias e gradientes podem ser dominados por variáveis numericamente maiores.

Quando as variáveis estão em escalas muito diferentes, alguns modelos dão peso desproporcional à variável de número maior — mesmo que ela não seja mais importante de verdade pro resultado.

Exemplo do problema: m² de uma obra varia de 40 a 2000; número de banheiros varia de 1 a 5. Numa conta de distância (KNN), a diferença de m² domina completamente a diferença de banheiros, mesmo que banheiros importe bastante pro preço final.

Duas técnicas comuns

  • Normalização (min-max): reescala tudo pra ficar entre 0 e 1, com base no mínimo e máximo da variável.
  • Padronização (z-score): reescala pra média 0 e desvio padrão 1 — mais robusta quando existem outliers na base.

Categoria vira vetorvisual explicativo · animação leve

As categorias deixam de parecer uma ordem numérica que não existe.

Modelos só entendem número. Uma coluna de texto como "tipo de material" (concreto, madeira, aço) precisa virar número — mas transformar direto em 1, 2, 3 seria um erro, porque isso implicaria uma ordem que não existe de verdade (aço não é "maior" que madeira).

Redução de dimensionalidade

Muitas dimensões para poucasvisual explicativo · animação leve

A projeção preserva parte da estrutura, mas pode reduzir interpretabilidade.

Quando sua base tem muitas variáveis (dezenas ou centenas), duas coisas ruins costumam acontecer: o modelo fica mais lento, e a "maldição da dimensionalidade" (folha 09) faz a noção de "parecido" perder sentido — com tantas dimensões, tudo parece igualmente distante de tudo. Redução de dimensionalidade é o processo de comprimir muitas variáveis em poucas, mantendo o máximo possível da informação original.

PCA (Análise de Componentes Principais)

É a técnica mais usada. Em vez de escolher quais variáveis descartar, o PCA combina variáveis correlacionadas entre si em um número menor de "componentes" novos, que capturam a maior parte da variação dos dados originais.

Construção civil

Muitas medidas por obra

30 variáveis registradas por obra (m², número de andares, tipo de fundação, prazo, número de fornecedores, etc.), várias correlacionadas entre si. PCA reduz isso a poucas componentes principais, facilitando agrupar obras parecidas (folha 12) sem ruído de variáveis redundantes.

Varejo

Comportamento de navegação no site

Centenas de variáveis de clique e navegação por cliente — reduzir pra 2 ou 3 dimensões permite visualizar padrões de comportamento num gráfico simples, algo impossível de "ver" com 200 variáveis soltas.

bom saber — reconhecer quando aplicar

Até onde vale ir

  • Precisa saber: que essa técnica existe, que problema ela resolve (muitas variáveis correlacionadas, lentidão, dificuldade de visualizar) e que PCA é a ferramenta padrão.
  • Bom saber: que depois do PCA você perde a leitura direta de "qual variável original pesou mais" — as componentes novas são combinações, não têm nome de negócio óbvio. Isso reduz a explicabilidade (folha 31).
  • Não precisa ir atrás: a álgebra linear por trás (autovalores, autovetores). Pra aplicar, basta saber chamar a função pronta e escolher quantas componentes manter.
Quando usar: antes de K-means ou KNN com muitas variáveis, ou quando quiser visualizar dados complexos num gráfico 2D/3D. Se sua base já tem poucas variáveis (menos de 10-15) e nenhuma delas é muito redundante, provavelmente você nem precisa disso ainda.

Redes neurais (introdução)

Camadas de transformaçõesvisual explicativo · animação leve

Pesos combinam sinais; ativações introduzem não linearidade; o erro ajusta conexões.

Pra dados em tabela (planilha de vendas, orçamento de obra), boosting e Random Forest quase sempre resolvem melhor e mais rápido. Redes neurais valem a pena quando o dado é imagem, texto, áudio ou vídeo — coisas que não cabem bem numa planilha.

A ideia central, sem fórmula

Uma rede neural é uma sequência de "camadas" de contas simples empilhadas. Cada camada aprende a reconhecer padrões um pouco mais complexos que a anterior — a primeira camada pode reconhecer bordas numa foto, a próxima reconhece formas, a próxima reconhece "isso parece uma trinca".
